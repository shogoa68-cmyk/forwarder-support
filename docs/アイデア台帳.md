## 🟠 中：あると便利・品質向上

- [ ] **Excel / CSV との双方向連携**
  - 背景: サーチャージ案内・顧客リスト等のデータは Excel で管理・配布されることが多く、アプリへの取り込みや逆方向の出力ニーズがある
  - 現状: 見積明細の Excel 出力は実装済み（SheetJS）。インポート機能は未実装
  - 設計方針:
    - **短期 — テンプレート方式**: サーチャージマスター・顧客マスター実装時にダウンロード用テンプレート Excel と読み込み機能をセットで作る。列が少ない初期段階はこれで十分
    - **中期 — 列マッピング方式**: 任意の Excel/CSV をアップロード → 列ヘッダーをアプリのフィールドにドロップダウンで対応付け → プレビュー確認 → 一括インポート。船会社ごとに微妙に異なる書式にも対応可能（Notion/Airtable と同じ考え方）
  - 想定する連携場面:
    - サーチャージ一括登録（船会社から届く Excel 添付をそのまま取り込む）
    - 顧客マスター一括登録
    - 案件一覧のエクスポート（案件管理を Excel で並行管理したい場合）
  - 注意点: 列順・列名のズレによる読み込みエラー対策・バリデーションを厚く作ることが重要
  - 前提: サーチャージマスター・顧客マスターの実装後に着手
  - 優先度: マスター実装と同タイミングで短期版を作るのが自然

- [ ] **案件のコピーから新規案件を作成する機能**
  - 背景: 類似した条件の案件を繰り返し作成するケースが多く、毎回ゼロから入力するのは非効率
  - 期待する動作:
    - クラウド案件一覧（またはプレビューモーダル）から「コピーして新規作成」ボタンを押すと、その案件の全データ（条件・明細・備考等）をコピーした新規案件が開く
    - コピー元の案件名に「(コピー)」等のサフィックスを付けて区別
    - コピー後はクラウド未保存の状態で編集可能（保存ボタンで別案件として登録）
  - 技術メモ:
    - `gatherAllData()` でデータ取得 → `_applyQuoteData()` で適用、という既存フローを流用できる
    - クラウド保存済み案件のコピーは `quote_presets` からデータを読み込んで `id` を null にして再保存
  - 前提: クラウドプリセット保存機能（実装済み）
  - UI候補: クラウドプレビューモーダルのヘッダー or アクションボタン列に追加

## 🟡 低：将来アイデア・検討中

- [ ] **見積タブの3ページ構成リデザイン**
  - 背景: 現状は条件・明細・統計が1画面に詰まっており、特にスプリットビューで窮屈
  - 構成案（合意済み）:
    - **1ページ目（ダッシュボード）**: 簡易統計（粗利率・合計金額推移など）、進捗管理（受注/失注/保留）、見積検索・一覧、新規見積作成ボタン
    - **2ページ目（条件入力）**: 輸送モード・引き合い条件・貨物・物量情報・マイルストーン
    - **3ページ目（明細入力）**: 現在の見積テーブル（費用明細・プレビュー・出力）
  - メリット:
    - 各ページが1つの役割に集中できる
    - 1ページ目は Supabase 連携後に統計・検索が自然に載る器になる
    - 2→3ページへの遷移で「条件確定 → 明細入力」のフローが明確になる
  - 前提: Supabase 基盤（統計・検索に必要）が整うタイミングで設計・実装

- [ ] **輸出+輸入セット対応（方向・マイルストーン）**
  - 背景: 輸出→輸入・輸入→輸出をワンセットで扱う案件が多数ある
  - 検証結果（合意済み）:
    - 輸送モード（FCL/LCL/Air）自体は方向に依存しないため変更不要
    - 費用カテゴリも既に輸出・輸入の両方が選択可能
    - **対応が必要な箇所は3箇所のみ**:
      - `index.html`: 方向ボタンに「↔️ 輸出+輸入」を追加
      - `conditions.js`: `_applyZoneLabels()` の 'both' 対応
      - `quote-email.js`: `dirMap` に `both: '輸出+輸入'` を追加
    - **マイルストーンは A案（2段表示）を推奨**: 輸出バーと輸入バーを縦に並べる。`renderQuoteMilestones` を export / import で2回呼ぶ設計。6ゾーン1本より業務実態に合う。
  - 変更量: 方向ボタン対応は小、マイルストーン2段表示は中

- [ ] **Supabase連携：行挿入パネル（行パターン＋サブコン別モジュールの統合UI）**
  - 背景:
    - 既存の「行パターン登録」はlocalStorage保存でチーム共有できない
    - サブコン別モジュール挿入（アイデア）と統合することで1つの「＋ 挿入」パネルに集約できる
  - 設計方針（合意済み）:
    - **2つは競合しない。同じパネルにタブ分けして共存させる**
    - 「保存パターン」タブ: 手動で名前をつけて登録した定型セット（現在の行パターン機能をSupabaseに移行）
    - 「サブコン別」タブ: 案件蓄積から自動生成。保存ステップ不要・直近案件が自動的に最新版になる
    - 両タブとも Supabase に保存 → チーム全員で共有・閲覧・挿入できる
  - 使い分け:
    - 保存パターン → 「社内標準セット」「航空基本セット」など意図的に管理したいテンプレ
    - サブコン別 → 「先月 ABC Trucking で使った費用をベースにしたい」など実績単価の流用
  - UIイメージ:
    ```
    「＋ 挿入」パネル
    [保存パターン] [サブコン別]
    ─────────────────────────
    保存パターンタブ:
      標準輸出セット（8行）
      航空基本セット（5行）
    サブコン別タブ:
      ABC Trucking（最終: 2026-06-01, 5行）
      横浜税関代行（最終: 2026-05-30, 3行）
    ```
  - 前提: Supabase への見積保存機能・認証基盤が先に必要

## 🔵 検討：方針・仕様確認が必要なもの

- [ ] **Supabase連携：案件蓄積によるマスター自動生成＋受注/失注分析**
  - 背景: 案件を貯めるだけで顧客・サブコン・費用項目の「マスター」が自然と出来上がる設計にしたい
  - 設計方針（合意済み）:
    - マスターテーブルは別途作らない。案件データへの集計クエリがマスター代わり
    - 顧客マスター: `quotes` の customer_name を DISTINCT 集計
    - サブコンマスター: `quote_rows` の subcon_name を DISTINCT 集計
    - 費用項目マスター: `quote_rows` の name を DISTINCT 集計＋平均単価・使用回数付き
    - 案件が増えるほど精度が上がる（50案件で顧客20社・費用項目80件相当のマスターが生える）
  - 活用例:
    - 品目入力時のオートコンプリート＋前回単価の自動補完
    - モジュール挿入のサブコン検索元（別アイデアと連携）
    - 顧客別・サブコン別のコスト傾向分析
  - 残課題: 表記ゆらぎ（例: "通関料" vs "通関費用"）は別途名寄せ作業が必要
  - 将来拡張: 使用頻度の高い項目を「正式マスター」に昇格させる仕組みを後付け可能

- [ ] **Supabase連携：受注/失注率の分析**
  - 背景: 見積の結果（受注・失注・保留）を記録することで勝率・傾向を把握したい
  - 想定する分析軸:
    - 顧客別・担当者別・ルート別・輸送モード別の受注率
    - 粗利率帯別の受注率（安い見積が取れているか）
    - 失注理由の傾向（価格・リードタイム・サービス等）
    - 月次・四半期の受注金額推移
  - 必要な追加フィールド: `status`（受注/失注/保留/提出済み）, `lost_reason`, `won_at`
  - 前提: Supabase への見積保存機能が先に必要


- [ ] **AIエージェントによるブッキング自動化**
  - 背景: フォームでブッキング依頼を受け付け、AIエージェントが各船会社ポータルでブッキングを完了させる
  - 段階的な実装方針（合意済み）:
    - **Stage 1**（実現しやすい）: AI が依頼内容を整形 → 各社フォームの入力値を自動生成 → 人が確認してワンクリック送信。現状の SI 作成の延長。入力作業の9割を省ける
    - **Stage 2**（中難度）: 公式 API 対応の船会社（Maersk・CMA CGM 等）のみ完全自動化
    - **Stage 3**（高難度・将来）: Claude Computer Use + Playwright で残りの船会社ポータルを自動操作
  - 現実的な壁:
    - 17社分の個別実装が必要（構造がすべて異なる）
    - CAPTCHA・2FA・アンチbot対策
    - ポータルUI変更による保守コスト
    - 各社利用規約の確認が必要（自動アクセス禁止の場合あり）
  - 前提: Supabase への依頼データ保存・お気に入りリンク（各社URL管理）が先に必要


- [ ] **将来構想：実務メニューと見積メニューの入り口分離**
  - 背景: 役割が異なるユーザー（オペレーション vs 営業/価格担当）が同じポータルを使うなら入り口を分けた方が使いやすい
  - 設計方針（合意済み）:
    - 実務メニュー: Booking/SI・スケジュール・お気に入りリンク・書類・通関・知識参考
    - 見積メニュー: 見積作成・費用計算・プレビュー/出力・受注/失注管理
    - データ層（Supabase）は共有: 顧客・サブコン・費用項目・お気に入りリンク・為替レート
  - メリット:
    - 実務側は軽快に・モバイル対応優先
    - 見積側は情報量を確保
    - 両側で同じマスターデータを参照できる
  - 前提: Supabase基盤の整備・マスター自動生成・お気に入り共有が揃ってから

- [ ] **お気に入り共有機能（船会社メニューのブックマーク）**
  - 背景: 船会社ごとに異なるポータルのレイアウトから目的のメニューを毎回探すのが非効率
  - 設計方針（合意済み）:
    - Supabaseに船会社別リンクを保存（carrier_name / label / category / url / note）
    - アプリ内で船会社・カテゴリでフィルターして一覧表示
    - チームメンバー全員が追加・編集できる（Supabase RLS で制御）
  - UIイメージ: 船会社タブ＋カテゴリフィルター＋リンク一覧＋追加ボタン
  - 配置候補: 既存「🚢本船・航路」タブへの組み込み or 新タブ
  - 前提: Supabase認証基盤が必要

- [ ] **顧客マスターの新設**
  - 背景: 現在の見積フォームには「会社名」「担当者名」しかなく、メールアドレス・部署・電話番号等が管理できていない。サーチャージ通知の宛先管理や見積フォームとの連動を考えると情報が不足している
  - 管理項目（確定）:
    - `customer_code`: アプリ内独自の管理番号（例: `C-001`）。社内基幹システムにすでにコードが存在する場合はそちらに合わせる形も可
    - `company_name`: 会社名（既存 `qf-customer` と連動）
    - `department`: 部署名（宛名出力に使用）
    - `person_name`: 担当者名（既存 `qf-person` と連動）
    - `email`: 担当者メールアドレス（通知宛先の自動セット）
    - `phone`: 電話番号
    - `note`: 備考（自由記述）
    - `created_by` / `updated_at`
  - 意図的に除外した項目:
    - 輸送モード・インコタームズ・取引通貨・支払条件 → 当面不要
    - 通知フラグ → 将来の「定期案内」機能側で管理する（顧客マスターには持たせない）
  - 顧客コードの方針:
    - 基本はアプリ内で独自採番（基幹システムへの依存を避ける）
    - 既存の基幹システムコードがある場合は `customer_code` にそれを使う形でも運用可
    - 基幹システムとの自動連携（API同期等）は将来拡張として検討
  - 見積フォームとの連動（将来）:
    - `qf-customer` 入力時に顧客マスターをサジェスト → 選択すると担当者・部署・メールが自動セット
    - 見積保存時に `customer_code` も一緒に記録し、案件←→顧客の紐付けを確立
  - テーブル案: `customers { id, customer_code(unique), company_name, department, person_name, email, phone, note, created_by, updated_at }`
  - RLS: `is_team_member()` で読み書き制御
  - 前提: Supabase 認証基盤（実装済み）
  - 依存: 「料金改定・サーチャージ通知管理」の宛先グループ機能と連動

- [ ] **料金改定・サーチャージ通知の管理・簡素化**
  - 背景: 客先への料金改定やサーチャージ導入の案内を定期的に行っているが、宛先の選定・文面作成・案内履歴の管理がすべて手作業で工数がかかる。抜け漏れや二重案内も発生しやすい
  - 実装したい機能（優先度順）:
    - **② 顧客グループ別の宛先管理**
      - 顧客マスターに輸送モード属性（FCL / LCL / AIR、複数可）を持たせる
      - 通知作成時に「FCL向け」「LCL向け」等を選ぶと該当顧客を自動抽出・一覧表示
      - 通知対象から除外したい顧客をチェックで外せる（個別調整）
    - **③ 通知履歴の記録**
      - 誰に・いつ・何の改定を・誰が案内したかを記録
      - テーブル案: `surcharge_notifications { id, surcharge_ids[], customer_ids[], notified_at, notified_by, note }`
      - 顧客一覧で「最終案内日」「案内済みサーチャージ」をひと目で確認できる
      - 「この顧客にはまだ案内していない」の抽出フィルターも対応
    - **④ 通知テンプレート管理**
      - 「GRI案内」「PSS導入」「料金改定のお知らせ」等の定型テンプレートを登録・管理
      - テーブル案: `notification_templates { id, name, subject, body_template, created_by }`
      - テンプレート本文にプレースホルダ（`{{carrier}}` `{{charge_name}}` `{{amount}}` `{{valid_from}}` 等）を埋め込み、サーチャージマスターの内容を差し込んで文面生成
      - 生成した文面はクリップボードコピー or メールクライアント起動（`mailto:`）で利用
  - UX の流れ（想定）:
    1. サーチャージマスターで「今回案内する改定」にチェック
    2. 通知テンプレートを選択
    3. 宛先グループ（FCL/LCL等）を選択 → 顧客一覧が自動抽出・確認
    4. 文面プレビューを確認してコピー or 送信
    5. 案内完了後に通知履歴として自動記録
  - 前提: 顧客マスター・サーチャージマスター・Supabase 認証基盤
  - 依存: 上記「マスタータブ新設」「サーチャージマスター」の実装後

- [ ] **マスタータブの新設（繰り返し登場する要素の一元管理）**
  - 背景: 案件が蓄積されると顧客・費用項目・サブコン・サーチャージなど繰り返し登場する要素が増える。現状はメール・社内サーバー・個人PC等に散在しており管理方法が統一されていない
  - **最優先: サーチャージマスター**
    - 各船会社・LCLキャリアーからのサーチャージアナウンスを一元管理
    - 登録項目（案）: `carrier`（船会社/キャリア名）/ `charge_name`（サーチャージ名）/ `amount`（金額）/ `currency`（通貨）/ `unit`（単位: per W/M, per B/L 等）/ `valid_from`〜`valid_to`（適用期間）/ `transport_mode`（FCL/LCL/AIR）/ `note`（備考・アナウンス原文リンク等）/ `source`（情報ソース: メール/公式サイト等）/ `created_by` / `updated_at`
    - 機能:
      - 一覧表示（会社別・期間別フィルター）
      - 新規登録・編集・削除（チームメンバー全員可）
      - 有効期限切れのアナウンスをグレーアウト or アーカイブ
      - 将来: 見積作成時にサーチャージマスターから行を自動提案・挿入
    - 現状課題: メール添付・社内サーバー保管・個人管理が混在しており参照先が定まっていない
  - **将来拡張（マスタータブに追加予定の要素）**:
    - 顧客マスター（顧客名・担当者・住所・インコタームズ傾向等）
    - サブコン単価マスター（よく使うサブコンの標準費用）
    - 費用項目マスター（案件蓄積からの自動生成 → 既存アイデア「Supabase連携：案件蓄積によるマスター自動生成」と統合）
  - UI配置: ポータルのタブバーに「🗂 マスター」タブを新設。サーチャージ・顧客・サブコン等をサブタブで切り替え
  - 技術検討事項:
    - テーブル: `surcharge_master { id, carrier, charge_name, amount, currency, unit, valid_from, valid_to, transport_mode, note, source, created_by, updated_at }`
    - RLS: `is_team_member()` で読み書き制御
    - 有効期限管理: `valid_to < now()` で自動的に期限切れ判定
  - 前提: Supabase 認証基盤（実装済み）
  - 優先度メモ: サーチャージマスターは他のマスターより実務インパクトが高いため先行実装を推奨

- [ ] **案件の共同編集対応（Presence + フィールドロック）**
  - 背景: 複数人が同じ見積を同時に開いて編集するケースがある。現状は後勝ち（最後に保存した人の内容で上書き）になりデータ消失リスクがある
  - 段階的な実装方針:
    - **フェーズ1 — 保存競合の検知・警告**（前提・最小限）:
      - 保存時に `updated_at` を比較し、自分が最後に読み込んだ時刻より新しければ「他のメンバーが更新しました。上書きしますか？」と警告
      - Supabase Realtime で `quote_presets` の変更をサブスクライブ → 他者の保存をリアルタイム通知
    - **フェーズ2 — Presence によるフィールドロック**（本命）:
      - Supabase Realtime の **Presence** チャンネルで「誰が今どの案件を開いているか・どの欄にフォーカスしているか」をブロードキャスト
      - 他メンバーが編集中のフィールドは入力不可（またはハイライト警告）にしてロック

    - **Presence チャンネル**: `supabase.channel('quote:{preset_id}').track({ user, field })` で実装可能
    - **ロックの粒度**: 明細行単位 or フィールド単位（細かいほど競合が減るが実装が複雑）
    - **切断時のロック解放**: ブラウザを閉じた場合の自動解放をどう保証するか（Presence は接続切断で自動解放されるため原理上は問題なし）
  - 前提: Supabase 認証基盤・クラウドプリセット保存（実装済み）、フェーズ1 完了後にフェーズ2 へ
  - 依存: データ構造の見直し（`data jsonb` 一括保存 → フィールド/行単位保存への移行）が先決

    - **② コード・事業ごと売却**: 物流テック系スタートアップ・SIer・大手フォワーダー（ヤマトHD・日本通運・近鉄エクスプレス等の新規事業部門）への譲渡
    - **③ 業界団体への提供**: JIFFA（日本国際貨物取扱業協会）等が会員向けツールとして採用。収益より業界貢献・名声という形になりやすい
  - このアプリの強み: 日本の中小フォワーダーの業務フローに特化しており、海外汎用ツール（CargoWise 等）が手を出しにくい領域
  - 今から意識しておくべきこと:
    - **技術面**: 個人の API キー・メールアドレスをコードから完全分離 / マルチテナント対応（現状は1チーム想定）/ ドキュメント整備
    - **ビジネス面**: 著作権・IP の整理（個人 or 法人で持つか）/ 利用規約・プライバシーポリシーの整備 / 競合調査（CargoWise, Logiper 等）
  - 方針メモ: 単体売却より「SaaS として育てた実績（契約社数・使用量）を持った状態で売る」方が交渉力が上がる。まずは複数チームで使える状態（マルチテナント化）を目指すのが先決
