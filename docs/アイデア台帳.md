- [x] **航空CW計算（計算タブ）に IATA 0.5kg 切り上げが効かず、2系統のCWが不一致** ✅ 2026-06-09
  - 背景: `calculator.js` の `calcAirCW()` は `Math.max(volW, weight)` を生値のまま表示（0.5kg切り上げなし）。一方 `cargo.js`／`shared/calc.js` の `airChargeableWeight()` は IATA準拠で 0.5kg 切り上げ済み。同一アプリ内でCWが食い違う
  - 対応: `calcAirCW()` を `SharedCalc.airChargeableWeight()` 経由に統一し、全経路で0.5kg切り上げに統一（commit `427c457`）

- [x] **複数航路（_routeEntries）登録時、件名・条件サマリが先頭航路しか出ない** ✅ 2026-06-09
  - 背景: `conditions.js getConditions()` は `_routeEntries[0]` だけを代表POL/PODに採用し、件名（`quote-pdf.js buildSubject`）も先頭航路のみ。明細行には各航路が出るのにヘッダーは1航路だけ、という不整合が客先資料で起きる
  - 対応: 御見積書 PDF の `buildSubject()` を複数航路対応に修正。複数登録時は航路ごとに全件併記（commit `427c457`）

- [x] **案件の共同編集対応（Presence + フィールドロック）** ✅ フェーズ1・2 実装済み（2026-06-09）／フィールド単位ロックは `data jsonb` 一括保存の見直しが前提のため保留
  - 背景: 複数人が同じ見積を同時に開いて編集するケースがある。現状は後勝ち（最後に保存した人の内容で上書き）になりデータ消失リスクがある
  - 段階的な実装方針:
    - **フェーズ1 — 保存競合の検知・警告**（前提・最小限）:
      - 保存時に `updated_at` を比較し、自分が最後に読み込んだ時刻より新しければ「他のメンバーが更新しました。上書きしますか？」と警告
      - Supabase Realtime で `quote_presets` の変更をサブスクライブ → 他者の保存をリアルタイム通知
    - **フェーズ2 — Presence によるフィールドロック**（本命）:
      - Supabase Realtime の **Presence** チャンネルで「誰が今どの案件を開いているか・どの欄にフォーカスしているか」をブロードキャスト
      - 他メンバーが編集中のフィールドは入力不可（またはハイライト警告）にしてロック
      - 編集中ユーザー名をフィールド上にツールチップ表示（例: 「🟢 田中さんが編集中」）
  - 技術検討事項:
    - **データ構造の変更が必要**: 現状は quote 全体を1つの `data jsonb` ブロブで保存している。フィールドロックを実装するにはフィールド単位（または行単位）での保存に設計変更が必要
    - **Presence チャンネル**: `supabase.channel('quote:{preset_id}').track({ user, field })` で実装可能
    - **ロックの粒度**: 明細行単位 or フィールド単位（細かいほど競合が減るが実装が複雑）
    - **切断時のロック解放**: ブラウザを閉じた場合の自動解放をどう保証するか（Presence は接続切断で自動解放されるため原理上は問題なし）
  - 前提: Supabase 認証基盤・クラウドプリセット保存（実装済み）、フェーズ1 完了後にフェーズ2 へ
  - 依存: データ構造の見直し（`data jsonb` 一括保存 → フィールド/行単位保存への移行）が先決

    ```sql
    ALTER TABLE allowed_emails ADD COLUMN role text NOT NULL DEFAULT 'member';
    CREATE OR REPLACE FUNCTION get_my_role()

      SELECT role FROM allowed_emails WHERE email = auth.email()
    $$ LANGUAGE sql SECURITY DEFINER;
    ```
