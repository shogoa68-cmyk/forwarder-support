## 🎯 プロジェクト概要

### 何を作っているか
フォワーダー（貨物輸送の手配をする会社）の担当者向け**業務支援Webポータル**。
ブラウザだけで動く（ビルド不要・Vanilla JS）。

### 2つの目的

| 目的 | 今の痛み | このポータルで解決すること |
|---|---|---|
| **見積支援** | 見積書を作るのに時間がかかる・属人化している | 運賃・サーチャージをリアルタイム参照しながら見積書を作れる |
| **オペ支援** | 船会社・港のルール情報がバラバラ | 必要な情報を一か所で検索・参照できる |

### ターゲットユーザー
- **Phase A〜B**：社内フォワーダー担当者
- **Phase C〜**：一般公開（SaaS化・収益化）

---

## 📦 フェーズ定義

### Phase A（現在）：外部公開情報の集約・社内実用化

**最優先機能：海上FCL 輸出　運賃・サーチャージ参照パネル**

```
ユーザーが仕向け地・船会社・コンテナ種別を選ぶ
  ↓
該当する Ocean Freight + サーチャージ一覧が自動表示
  ↓
見積書にワンクリックで反映
```

**Phase A で作るもの（優先順）**
1. 運賃・サーチャージ参照パネル（海上FCL・輸出から）
2. 載せ幅ルール管理（属人化解消）
3. 見積データの保存・分析基盤（後で活かせる設計）
4. 船会社・港情報データベース（オペ支援）

---

### Phase B：社内本格運用
- GitHub Private + 有料プランへ移行
- 社内DBサーバー連携
- 見積分析・受注/失注管理
- 海上LCL対応

---

### Phase C：一般公開・収益化
- マルチテナント対応（複数社が使える）
- SaaS化（月額課金）
- 航空貨物モード追加

## 🗄️ データ設計方針

### 基本原則：最初から移行を想定した設計

```
【Phase A】  Googleスプレッドシート + GAS
                ↓ （骨子完成後）
【Phase B】  社内DBサーバー（会社が用意）
                ↓ （SaaS化時）
【Phase C】  マルチテナント対応DB
```

### データアクセス層（APIレイヤー）を必ず1か所に集める

> ポイント：データの読み書きをする処理を1か所にまとめておく。
> 将来DBが変わっても、その1か所だけ書き換えればフロントエンド側は触らなくていい。

```javascript
// Phase A
const dataSource = new GASDataSource(GAS_URL);

// Phase B（ここだけ変える）
const dataSource = new InternalDBDataSource(API_ENDPOINT);
```

### 運賃マスタに持つ項目（案・要確定）
- 船会社名
- 仕向け地（Port of Discharge）
- コンテナ種別（20' / 40' / 40'HC）
- Ocean Freight（運賃）
- サーチャージ種別・金額（BAF / CAF / PSS / GRI / EBS など）
- 有効期限（Valid From / To）
- 備考

## 💬 フィードバック収集の仕組み

### 概要
アプリを操作しながら気づいたことを即座に記録し、Googleスプレッドシートに蓄積する。
溜まったフィードバックはチャットで話し合い、Claude Codeで一括実装する。

### 流れ
```
アプリを操作中に「ここ直したい」「この機能欲しい」と気づく
  ↓
セクションの 💬 ボタンをクリック
  ↓
種別（バグ／改善／新機能）と内容だけ入力して送信
  ↓
Googleスプレッドシートに自動蓄積
  ↓
チャットで話し合い → Claude Codeで一括実装
```

### UIの仕様
- 各セクションのヘッダー付近に 💬 ボタンを設置
- **セクション名は自動取得**（ユーザーが入力する必要なし）
- ユーザーが入力するのは「種別」と「内容」だけ

### スプレッドシートの列構成
| 列 | 内容 | 取得方法 |
|---|---|---|
| 日時 | 送信した日時 | 自動 |
| セクション | どの画面・機能への意見か | ボタン設置場所から自動取得 |
| 種別 | バグ／改善／新機能 | ユーザーが選択 |
| 内容 | 気づいたこと・要望 | ユーザーが入力 |
| ステータス | 未対応／対応中／完了 | 「未対応」で自動セット |

### ⚠️ 開発用機能として管理する（フィーチャーフラグ）

> 💡 フィーチャーフラグとは：機能のON/OFFを1行で切り替えられるスイッチ。
> 将来の本番リリース時に、コードを大きく変えずに機能を消せる。

```javascript
// config.js — ここを変えるだけで全フィードバックボタンが消える
const APP_CONFIG = {
  DEV_MODE: true,          // false にすると全ボタンが非表示
  FEEDBACK_SHEET_URL: ""   // GASのエンドポイントURL
};
```

```javascript
// 呼び出し側
if (APP_CONFIG.DEV_MODE) {
  initFeedbackButtons(); // DEV_MODE=false なら何も起きない
}
```

**将来の削除作業：**
1. `config.js` の `DEV_MODE` を `false` にする → ボタンが全消え
2. （余裕があれば）`class="dev-feedback-btn"` のHTML要素を削除

## 🔒 セキュリティ方針

| リスク | 対策 |
|---|---|
| APIキー漏洩 | GASをサーバー代わりにしてキーを隠す |
| 不正アクセス | Googleアカウント認証（Phase B〜） |
| 通信傍受 | Netlify の HTTPS で対応済み |
| コードへの直書き | `.env` を使い `.gitignore` に追加する |

### ⚠️ 絶対やってはいけないこと
```javascript
// ❌ ダメ（Publicリポジトリでは世界中にバレる）
const API_KEY = "AIzaSyXXXXXXXXXXX";

// ✅ 正解
const API_KEY = process.env.API_KEY; // 環境変数で管理
```
